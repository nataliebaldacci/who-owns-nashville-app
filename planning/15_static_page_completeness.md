## Implementation — Owner Profile Page (`/owner/{cluster_id}/`) ✅ DONE

### 1. Remove SOS `<details>` expand/hide wrapper ✅ implement
The section already renders `open` by default. The expand/hide adds friction with no benefit.
Replace `<details class="sos-details" open>` / `<summary>Georgia SOS …</summary>` with a flat
`<p class="profile-section-label">GEORGIA SOS …</p>` consistent with the "COUNTY TAX PARCEL"
section above it.

### 2. GA SOS direct links per entity ✅ implement
The parcel panel already has this (via `sos_business_id`); the owner page has none.

**Data change:** `fetch_sos_details_batch()` must also pull `sos_business_id` per entity.
Aggregate into a list of `{name, sos_business_id}` pairs (one per unique entity in the cluster
that has a SOS match).

**Display:** Render a small "SOS ↗" link next to each entry in "Owner Names on record" (see §5)
where a `sos_business_id` is available. Format:
`https://ecorp.sos.ga.gov/BusinessSearch/BusinessInformation?businessId={id}`

### 3. Add principal office location ✅ implement
`sos_principal_city` and `sos_principal_state` are already on `owner_entities` — unused.
Show a single "Principal office" field in the SOS section, aggregating unique city/state pairs
across entities in the cluster (skip duplicates, skip blank). Example:
> Principal office: Scottsdale, AZ · Atlanta, GA

Flag out-of-state entries with a subtle indicator (same `badge-state` style used on leaderboard
for `foreign_state`).

### 4. Rename "Names on record" → "Owner Names on record" ✅ implement
Heading text only. Also update the quicknav anchor link text (`<a href="#aka">names on record →</a>`).

### 5. SOS link per name in "Owner Names on record" ✅ implement
Currently `alt_names` is a plain list of strings. To support per-name SOS links, change the
data structure passed to the template to `[{name, sos_business_id_or_null}]`. Query:
join `owner_entities` on `cluster_id`, group by `owner_name_norm`, take max `sos_business_id`.

Render: if `sos_business_id` present, link the name; otherwise plain text.

### 6. Mailing addresses — scrollbox with total count ✅ implement
Replace the "(first 8)" truncation note with a CSS scrollbox:
- Show all addresses (remove the `:8` slice — or keep a higher cap, e.g. 20).
- Wrap `<ul class="address-list">` in a `<div class="scroll-box">` that gets `max-height` + `overflow-y: auto` in CSS.
- Move the count into the `<dt>` label: "Mailing addresses (14)".

### 7. Registered agents — scrollbox + full count ✅ implement
Same treatment as addresses. Raise the cap from 10 to something higher (e.g. 20), use a
scrollbox, show count in the `<dt>` label.

### 8. Related owners — show specific shared address ✅ implement
`build_cluster_related()` currently uses the generic string `"Shared address"` for address
connections. The actual address is available as the loop variable `addr`. Change to:
```python
reason = f"Shared address: {addr}"
```
Zero data cost — the address is already in memory.

### 9. Related owners — don't link single-parcel clusters ✅ implement
Owner pages are only generated for `parcel_count >= 2`. The template currently links all
related owners unconditionally. Add a conditional:
```jinja
{% if r.parcel_count >= 2 %}
  <a href="/owner/{{ r.cluster_id }}/">{{ r.primary_name | e }}</a>
{% else %}
  {{ r.primary_name | e }}
{% endif %}
```

### 10. Parcels table — map deep link per row ✅ implement
Each parcel row should link to the map with that parcel's card open. Requires two changes:

**app.js:** Add `?parcel={county}/{parcel_id}` deep link handling alongside the existing
`?cluster=` handler. On map load, if `parcel` param is present, call
`loadParcel(county, parcel_id)` after tiles are ready. This also needs a map fly-to
(the parcel lat/lon is returned by `/api/parcel/`).

**Template:** Add a map column to the parcel table:
```html
<a href="/?parcel={{ p.county }}/{{ p.parcel_id | urlencode }}" class="map-link">map →</a>
```

**Data:** `p.parcel_id` and `p.county` are already in `fetch_parcels_batch()` output.

### 11. Officers — add to owner profile ✅ implement
`sos.officers` is queried in the live API but never in `build_static_pages.py`. Officers are
one of the highest-signal fields for identifying networked ownership.

**Data change:** Add `fetch_officers_batch(conn, cluster_ids)` — join `owner_entities`
→ `sos.entities` (via `sos_control_number`) → `sos.officers`. Return
`{cluster_id: [{role, name, city, state}, ...]}`, deduped, capped at 20 per cluster.

**Display:** New section in the owner profile between SOS and Neighborhood Breakdown:
```
OFFICERS / PRINCIPALS   [*]
- Manager: John Smith — Atlanta, GA
- Member: Doe Holdings LLC — Wilmington, DE
```
Show city/state when present. Skip blank names. If officer name appears in `linkable_agent_ids`
(same individual serves as RA for multiple clusters), note that connection.

## New Static Page Category: Shared Address Pages ✅ DONE

### 16. `/l/addresses/{slug}/` pages ✅ implement (low build cost, high investigative value)

**Deviation from plan:** URL changed from `/address/{slug}/` to `/l/addresses/{slug}/`
to keep shared address pages under the leaderboard hierarchy (`/l/`). A backward-compat
alias at `/addresses/{slug}/` is also generated. nginx location block added for
`/addresses/` alias in both dev and prod configs.
`fetch_address_linkage()` already computes ~269 address groups (addresses shared by 2–10
clusters). A shared PO box or street address connecting multiple LLCs is itself a meaningful
network node — currently this signal is buried in individual owner profiles' "Related owners"
tables and invisible from any navigational entry point.

**Page format** — mirrors the Agent page pattern:
```
123 Peachtree St NW, Atlanta GA  [shared mailing address]
─────────────────────────────────────────────────────
Owners sharing this address: 5 clusters · 47 total parcels

| Owner               | Parcels | Flags       |
|---------------------|---------|-------------|
| SMITH HOLDINGS LLC  | 18      | CORPORATE   |
| PEACHTREE MGMT LLC  | 12      | CORPORATE   |
| ...                 |         |             |
```

**URL:** Slug from the address string, e.g. `/address/123-peachtree-st-nw-atlanta-ga/`.

**Index:** `/address/` — table of all shared addresses, sorted by cluster count desc.

**Navigation:** Link from the Registered Agents index page ("Shared Addresses" alongside
"Registered Agents" in the leaderboard subnav). Also link from Related Owners table on owner
profiles where `via` starts with "Shared address:".

**Link from owner profiles:** When a "Related owners" connection is via shared address, the
address text becomes a link to `/address/{slug}/` instead of plain text.

## Push for Later

### Officers → per-person pages (`/person/{slug}/`)
The same pattern as agents: if an officer (member/manager/principal) appears across multiple
clusters, pre-generate a page listing all their clusters. The `sos.officer_global_counts` table
already tracks this. Defer until officers are confirmed working in the owner profile (§11) and
we can evaluate how many linkable individuals exist.

### Agents index — agent address/city/state column
The agents index (`/l/agents/`) currently shows name, cluster count, total parcels. Adding
city/state of the agent would help users triage (local agent vs. out-of-state). `sos_registered_agent_address` is available but is a raw unstructured string — parsing city/state reliably
is the blocker. Defer until address parsing is cleaner or we join through `sos.registered_agents`.

### "Owner Persona" / automated characterization
Plan 09 §3 describes pre-calculating a hub location, county focus, and portfolio mix
characterization. Useful but not urgent — the raw data additions in this plan (principal office,
officers, SOS links) give investigators the inputs to form their own characterization. Revisit
after this pass is live.

### Assessment value totals
Rolling up DeKalb `totapr1` across cluster parcels gives "this owner controls ~$X in property."
Powerful but incomplete (Fulton values unavailable in the same way). Defer until either Fulton
values are sourced or we decide a clearly-labeled DeKalb-only figure is acceptable.

### Homestead exemption count on owner profile
"0 of 50 Fulton parcels have homestead exemption" = clear all-rentals signal. Fulton only.
Easy to compute (`excode IS NOT NULL` count). Low effort — could pull forward into this plan.
Left here pending confirmation that the data is clean enough.

### SOS commencement / formation date
`sos.entities.commencement_date` — when was this entity formed? Useful context (e.g., LLC
formed 6 months ago already owns 40 parcels). Defer — requires an additional join in
`fetch_sos_details_batch()` and the date data quality needs verification.

### Choropleth / corporate ownership layer on map
Stretch goal from plan 04. Not part of static pages — separate map feature.

### Network diagram on owner profile
Plan 09 explicitly defers this. The Related Owners table covers the same investigative need.
Do not revisit until the table proves insufficient.

### Parcel table — full-list scrollbox (>200 parcels)
Currently capped at 200 with a prose message to use the map. A scrollable DOM table for
300+ parcels is heavy. Current approach is acceptable. Revisit only if the map deep link (§10)
proves insufficient.

## Implementation Notes / Deviations

- **`via_items` data structure**: Related owners `via` field changed from a plain string
  to `via_items: [{text, url}]` list — enables per-reason hyperlinks (RA → `/agent/`,
  address → `/l/addresses/`). Template renders each item with optional `<a>` wrap.
- **Address cap raised**: Mailing addresses cap changed from 8 to 20 (scroll-box handles
  the visual length). RA cap raised from 10 to 20. Both use CSS `.scroll-box` class.
- **`_make_env()` helper**: All Jinja2 `Environment(loader=BaseLoader())` calls replaced
  with `_make_env()` which registers `urlencode` filter (needed for parcel map links).
- **`fetch_geo_data` / `fetch_county_geo_data`**: Now return `alt_names` (from
  `owner_names[2:4]`) and `connection_count: 0` placeholder (filled in by
  `_build_geo_section` from `cluster_connection_count` dict).
- **`fetch_agent_clusters`**: Now joins `mv_cluster_stats` to return
  `is_corporate`/`is_institutional` flags for the agent page cluster table.
- **`fetch_address_linkage`**: Now joins `mv_cluster_stats` to return
  `is_corporate`/`is_institutional` flags for address page cluster table.
- **3011 shared address pages** generated (2–10 cluster threshold from existing
  `fetch_address_linkage` query). This is more than the ~269 estimated in the research
  doc — the actual DB has more qualifying addresses.
- **Prod nginx config**: Added all missing static location blocks (`/l/`, `/owner/`,
  `/agent/`, `/agents/`, `/addresses/`, `/leaderboard/`) which were absent from the
  existing prod config placeholder.
